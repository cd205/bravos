- `Restart=always` + `RestartSec=15` matches the established `ibgateway.service` pattern (15s cooldown allows time for a Python startup that calls GCP Secret Manager).
- `User=chris_s_dodd` (non-root) — matches the existing pattern.
- `WantedBy=multi-user.target` — standard for non-graphical services.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Filesystem -> systemd (process env) | `/etc/bravos/env` content is read by systemd and injected into the service's process environment. A non-root user with write access to this file could plant a malicious BRAVOS_DB_PASSWORD or change TRADING_MODE silently. |
| systemd unit file -> ExecStart | Whoever can write to the .service file (or to the file ExecStart points at) executes code as `chris_s_dodd`. The .service files live in `infra/` (repo) and are symlinked into `/etc/systemd/system/` on the VM. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Information Disclosure | `/etc/bravos/env` containing BRAVOS_DB_PASSWORD | mitigate | env.example documents the required `sudo chown root:root /etc/bravos/env && sudo chmod 600 /etc/bravos/env` in the file header so the operator cannot miss it; verified via the `chmod 600` grep in acceptance criteria. The actual mode-change runs on bravos-vm1 in Plan 03. |
| T-08-02 | Tampering | `bravos-trading.service` ExecStart path | mitigate | The .service file is in `infra/` under git review; symlinking from `/etc/systemd/system/` keeps the canonical source under version control. Only root or `chris_s_dodd` can modify the symlink target; commit history provides an audit trail. |
| T-08-03 | Elevation of Privilege | `Requires=ibgateway.service` accidentally added | mitigate | Acceptance criteria explicitly require `grep -c "^Requires=" returns 0` for both .service files. RED-team rationale documented in task action (Requires= would let an attacker who can crash IB Gateway also force a daemon stop, opening a re-init window). |
| T-08-04 | Information Disclosure | Service env vars visible via `/proc/$PID/environ` to root and the service user | accept | Unavoidable consequence of `EnvironmentFile=`. On a dedicated single-user VM (bravos-vm1) where only `chris_s_dodd` and `root` have shell access, this is acceptable. Documented in RESEARCH Security Domain. |
| T-08-05 | Spoofing | systemd reads an env file written by an attacker who exploits any weakness in `chris_s_dodd`'s write path | transfer | Mitigation lives at OS level: `/etc/bravos/env` must be `root:root` mode 600 (Plan 03 operator step). env.example states this requirement. Beyond that, OS-level file permissions are the trust anchor — Phase 8 inherits the VM's hardening posture. |
</threat_model>
